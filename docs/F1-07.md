## Bug corretti

### 1. Negative lookahead `(?!...)` non supportato

Go's RE2 regex engine non supporta `(?!...)`. Il matcher
`path_regexp /admin/(?!_nuxt|favicon|manifest\.webmanifest).*`
crashava `caddy validate`:
```
Error: provision http.matchers.path_regexp: compiling matcher regexp
  /admin/(?!_nuxt|favicon|manifest\.webmanifest).*: invalid or unsupported
  Perl syntax: `(?!`
```

**Fix:** usare la first-match-wins di Caddy. `@admin_static` viene
valutato **prima** di `@admin_api`, quindi il secondo non ha bisogno
di escludere gli asset esplicitamente:
```caddy
@admin_api {
    path /admin/*
    method POST PUT PATCH DELETE
}
```
Se una POST arriva su `/admin/_nuxt/x.js`, `@admin_static` (GET-like
path match) non la cattura (vorrebbe GET/HEAD), ma è comunque coperta
da `@admin_api` (POST). Se GET su `/admin/_nuxt/x.js`, `@admin_static`
vince. Risolto senza lookahead negativo.

### 2. `file_server { precompressed br gzip }` inline

Caddy parser non accetta la forma inline con directive subcomandi —
serve multilinea:
```caddy
file_server {
    precompressed br gzip
}
```

Error: `Unexpected next token after '{' on same line, at
/etc/caddy/Caddyfile:132`.

LAN Caddyfile già lo scriveva corretto. PROD aveva la forma inline;
fixato.

### 3. `{env.VAR}` vs `{$VAR}` nelle site address

Caddy risolve le env var con due sintassi diverse:
- `{$VAR}` → shell-style, risolto al **config parse time**
- `{env.VAR}` → risolto a **request time**

Nelle site address (l'URL prima del `{`) serve la prima perché Caddy deve
conoscere l'hostname durante il parse (es. per emettere cert ACME).

Error: `subject does not qualify for certificate: '{env.MEMORYMESH_HOSTNAME}'`
(placeholder non espanso → Caddy vede la stringa letterale).

**Fix:** `Caddyfile.prod` usa `https://{$MEMORYMESH_HOSTNAME}` e
`email {$ADMIN_EMAIL}`.

### 4. `header_up X-Forwarded-For` / `X-Forwarded-Proto` ridondanti

Caddy ha pubblicato warning:
> Unnecessary header_up X-Forwarded-For: the reverse proxy's default
> behavior is to pass headers to the upstream

Non fatale, ma rumoroso. Rimossi da LAN + PROD (Caddy gestisce di default).

## Decisioni di design

### Standalone vs compose

F1-07 **non** usa `docker compose up caddy` perché il service `caddy` nel
compose ha `depends_on: api { condition: service_healthy }`, e l'immagine
`api` non è buildabile (manca `requirements-hashes.txt`, deferito a F2-01).

Alternativa: `docker run caddy:2-alpine` con network ad-hoc + stub Python
http.server con network-alias `api`. Il Caddyfile hardcoda
`reverse_proxy api:8000` → il stub DEVE chiamarsi `api` per matcha.

### No pull modelli / TLS real

`caddy validate` su Caddyfile.prod funziona anche senza ACME attivo:
passa env `MEMORYMESH_HOSTNAME=test.invalid.example` (TLD riservato) +
`ADMIN_EMAIL=admin@invalid.example`. Caddy valida la sintassi senza
tentare di emettere cert reali.

Test TLS live con Let's Encrypt staging → defer F8-04 (Tailscale) /
F12-11 (Caddy TLS profile-specific).

### Rate limit plugin commentato

I blocchi `rate_limit {...}` in entrambi i Caddyfile sono **commentati**:
il plugin `caddy-ratelimit` non è nell'immagine base `caddy:2-alpine`.
Richiederebbe xcaddy build → incremento complessità image. Tracked come
F12-11 (caddy-ratelimit via xcaddy, profile-specific limits).

## Gotcha incontrati

1. **Go RE2 vs Perl regex**: Go standard `regexp` usa RE2 → no `(?!...)`,
   no backreferences, no `\K`. Sempre verificare compatibilità regex in
   tool Go-based (Caddy, esbuild, Hugo).
2. **`{env.VAR}` site-address**: bisogna usare `{$VAR}` nelle site
   address, `{env.VAR}` va bene nelle directive value.
3. **MSYS path conversion**: Git-Bash converte `-v host:/container` in
   `-v host:C:/Program Files/Git/container`. Fix: `export
   MSYS_NO_PATHCONV=1` all'inizio dello script.
4. **grep `-i -F -q` combo crasha** su Git-Bash (Aborted). Usare `-qi`
   senza `-F`, ed escape manuale se serve regex-literal.
5. **Stub http.server** supporta solo GET (no HEAD). Il wait-loop deve
   usare GET `/` + match 3-digit code, non HEAD `/health`.
