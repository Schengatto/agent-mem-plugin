## Message contracts

Tutti i job hanno `v: int` (schema version) e `attempt: int` (retry count
incrementato dal worker). `extra='forbid'` impedisce che un publisher
aggiunga campi sconosciuti per sbaglio.

**EmbedJob**:
```python
{
  "v": 1,
  "attempt": 0,
  "obs_id": int,                                      # > 0
  "project_id": UUID,
  "content_kind": "observation"|"manifest_entry"|"vocab",
}
```

**DistillJob**:
```python
{
  "v": 1,
  "attempt": 0,
  "project_id": UUID,
  "trigger": "cron"|"manual"|"retry",
}
```

**TightenJob**:
```python
{
  "v": 1,
  "attempt": 0,
  "obs_id": int,
  "project_id": UUID,
  "reason": "capped_at_write"|"distill_cluster",
}
```

**DlqEnvelope** (pubblicato dal worker dopo MAX_ATTEMPTS):
```python
{
  "v": 1,
  "original_stream": str,                  # es. "mm:embed_jobs"
  "original_id": str,                      # Redis entry ID "1234567890-0"
  "payload": dict,                         # job originale serializzato
  "error_class": str,                      # es. "TimeoutError" (<=64)
  "error_message": str,                    # <=512
  "attempts": int,                         # > 0
  "first_failure_at": datetime,
  "last_failure_at": datetime,
}
```

## Pattern di serializzazione XADD

Redis Streams non supporta JSON nativo: ogni campo deve essere stringa.
Convenzione per publisher:

- `UUID` â†’ `str(uuid)` hex lowercase con trattini
- `int` â†’ `str(n)`
- `Literal` / enum â†’ string
- `list[str]` â†’ `",".join(...)` (nessun elemento contiene `,`)
- `datetime` â†’ ISO-8601 con timezone

**Mai** serializzare JSON annidato in un singolo field â€” splitting flat
semplifica `XRANGE` ispezione manuale. Esempio corretto per DLQ:

```python
# âś“ flat
{"payload.obs_id": "42", "payload.project_id": "...-..."}
# âś— nested
{"payload": '{"obs_id": 42, "project_id": "..."}'}
```

## Helper `ensure_stream_group`

```python
await ensure_stream_group(redis, stream, group, start_id="0")
```

- Idempotente: cattura `BUSYGROUP` al secondo call.
- `MKSTREAM`: crea lo stream se non esiste ancora (boot pulito).
- `start_id="0"` (default): il group legge anche i messaggi arretrati.
  Corretto per worker che riprendono dopo un restart â€” no message loss.
- `start_id="$"`: solo messaggi nuovi. Usare se il backlog Ă¨ indesiderato
  (es. worker "fresh" che non deve processare job pre-deploy).

Return: `True` se creato, `False` se giĂ  esistente (utile per log).

## Smoke test

`bash scripts/test-redis-streams.sh` esegue 12 verifiche:

1. Pydantic schemas happy path (EmbedJob/DistillJob/TightenJob)
2. `extra='forbid'` rifiuta campo sconosciuto
3. `obs_id > 0` validation
4. `ensure_stream_group` crea 3 stream + 3 group ex-novo (return True)
5. `ensure_stream_group` idempotency (return False)
6. `XINFO GROUPS` â€” group reale sui 3 stream
7. Round-trip XADD di un `EmbedJob` serializzato
8. `XREADGROUP` legge il messaggio
9. Pending list (`XPENDING`) == 1 prima di ACK
10. `XACK` ritorna 1 + pending list == 0 dopo ACK
11. DLQ: XADD envelope su 3 DLQ, `XINFO GROUPS` vuoto, `XLEN == 1`
12. `start_id='$'` â€” nuovo group non legge backlog pre-esistente

## Known issues â€” follow-up F1-02

Durante l'implementazione Ă¨ emerso un bug latente di `docker-compose.yml`:
il servizio `redis` non parte su Docker Desktop Windows per
`Permission denied` creando `/data/appendonlydir` (AOF). Combinazione di
`read_only: true` + volume ownership non-propagated + alpine redis user
UID 999.

Il test bypass disabilitando AOF/RDB nell'override compose (non serve
durabilitĂ  per uno smoke test). **Il bug in sĂ© va risolto in un task
follow-up** (F1-XX o fix PR su F1-02) prima del Gate M0 "make up OK":

- Opzione A: rimuovere `read_only: true` da redis (security downgrade).
- Opzione B: `user: "0:0"` esplicito (downgrade).
- Opzione C: init-container che pre-chowna `/data` + UID specifico.
- Opzione D: named volume con `driver_opts: { type: tmpfs }` per dev (ma
  perde persistenza).

Per la CI Linux il bug potrebbe non manifestarsi (volume semantics
diversa). Da validare con CI prima di promuovere.

## Note operative

incrementare `v` a 2 e supportare entrambe le versioni nei worker. Non
rimuovere mai un field pre-esistente senza una major bump dell'API Redis.

stuck in PEL oltre `MIN_IDLE_TIME`) arriva in F3-01 via `XAUTOCLAIM`
