# F1-06 — Ollama (pull script, env hardening, smoke test)

**Status:** ✅ done — 2026-04-22.

Ollama è **opt-in** (Profile B/C, privacy-strict). Profile A (default,
Gemini cloud) non avvia il container. F1-06 formalizza:

1. Script idempotente di pull modelli (`scripts/ollama-pull.sh`)
2. Smoke test che verifica profile-gating, env hardening, healthcheck
3. Makefile target + CI integration

## Deliverable

| File | Ruolo |
|------|-------|
| `scripts/ollama-pull.sh` | Pull idempotente di `OLLAMA_EMBED_MODEL` + `OLLAMA_DISTILL_MODEL`. Skip se modello già presente; `--embed-only`/`--llm-only`/`--dry-run` flags. |
| `scripts/test-ollama.sh` | 7 check: profile opt-in, `--profile ollama` boot, healthcheck, env hardening (3 var), `ollama list`, pull script syntax, dry-run. |
| `Makefile` | `ollama-check` (smoke test); `ollama-pull` riscritto per usare lo script. |
| `.github/workflows/ci.yml` | Job `ollama-smoke` dopo `compose-validate`. |

## Decisioni di design

### 1. Profile-gated (opt-in, non-default)

Default MemoryMesh è Profile A (Gemini cloud, `MEMORYMESH_LLM_PROVIDER=gemini`).
Il container `ollama` ha `profiles: [ollama]` in compose, quindi `make up`
non lo avvia. Solo:

```bash
docker compose --profile ollama up -d
```

attiva Ollama. Il primo test dello smoke test verifica esplicitamente che
`docker compose config --services` senza `--profile` NON elenchi `ollama` —
regression test per future modifiche al compose.

### 2. Pull idempotente via `ollama list`

Lo script esegue `ollama list` (output tabellare) e matcha la prima colonna
(`NAME`) con AWK `$1 == modello`. Se presente → skip. Se assente → `ollama
pull`. Safe su re-run (niente re-download multi-GB).

Il parser evita false positive su substring: `qwen3.5:9b` non matcha
`qwen3.5:9b-instruct`.

### 3. Env hardening (memoria)

`docker-compose.yml` setta tre env var critiche per footprint RAM:

- `OLLAMA_MAX_LOADED_MODELS=1` — mai più di 1 modello in RAM. Evita che
  Ollama tenga embed+LLM caricati insieme (risparmia ~5.5GB RAM picco).
- `OLLAMA_KEEP_ALIVE=5m` — unload automatico dopo 5 min idle. Critico per
  Profile C mini-PC 8GB (altrimenti OOM durante workload misto).
- `OLLAMA_HOST=0.0.0.0:11434` — bind sulla rete interna Docker per
  raggiungibilità dai worker.

Lo smoke test verifica tutte e tre via `printenv` dentro il container.

### 4. Parser env robusto (no `source .env`)

`.env` può contenere valori non-quotati con spazi:

```bash
DISTILLATION_CRON=0 3 * * *
```

`source .env` fallisce: bash interpreta `3` come comando. Il pull script
usa `sed -nE "s/^KEY=.../..."` per estrarre solo le due chiavi necessarie
(`OLLAMA_EMBED_MODEL`, `OLLAMA_DISTILL_MODEL`) senza evaluation del
contenuto. Più sicuro + più robusto.

### 5. Smoke test NON pulla modelli veri

`nomic-embed-text-v2-moe` è ~500MB, `qwen3.5:9b` è ~5.5GB. Troppo lento
e costoso per CI (timeout + bandwidth). Il test:

- Verifica `ollama list` risponde (output atteso: header `NAME`)
- Verifica `ollama-pull.sh --dry-run` con modelli non presenti → output
  atteso: `[dry-run] avrebbe eseguito: ollama pull ...`

Il pull vero resta responsabilità del developer/admin in setup locale via
`make ollama-pull`.

## Uso

### Setup Profile B (embed locale, LLM cloud)

```bash
# .env
MEMORYMESH_EMBED_PROVIDER=ollama
MEMORYMESH_LLM_PROVIDER=gemini     # LLM resta cloud

docker compose --profile ollama up -d ollama
make ollama-pull                    # pulla solo ciò che serve (em bed model)
# il LLM non viene pullato perché non è usato localmente (Gemini cloud)
# (nota: lo script attualmente pulla entrambi — usare --embed-only per
#  skippare il LLM fino a quando viene aggiunto un check provider nel
#  pull script. Follow-up minore.)
```

### Setup Profile C (full local)

```bash
# .env
MEMORYMESH_EMBED_PROVIDER=ollama
MEMORYMESH_LLM_PROVIDER=ollama

docker compose --profile ollama up -d ollama
make ollama-pull                    # pulla entrambi
```

### Flags pull script

```bash
scripts/ollama-pull.sh              # pull di entrambi (default)
scripts/ollama-pull.sh --embed-only # solo embed (Profile B)
scripts/ollama-pull.sh --llm-only   # solo LLM (edge case)
scripts/ollama-pull.sh --dry-run    # lista cosa farebbe, senza download
```

### Smoke test

```bash
make ollama-check                   # 7 check, senza download modelli
```

## Test coverage (7/7 pass locale)

1. Default profile non include `ollama` in `config --services`
2. `--profile ollama` include `ollama` in services
3. Boot `--profile ollama up -d ollama` → healthy entro 120s
4. `OLLAMA_MAX_LOADED_MODELS=1` settato dentro container
5. `OLLAMA_KEEP_ALIVE=5m` settato dentro container
6. `OLLAMA_HOST=0.0.0.0:11434` settato dentro container
7. `ollama list` risponde con header `NAME` (stato iniziale empty)
8. `ollama-pull.sh --dry-run` attraversa stato atteso (skip o `[dry-run]`)

(l'enumerazione include il syntax check bash -n come check-zero implicito)

## Follow-up identificati

1. **Pull condizionale su provider** (minor): oggi `ollama-pull.sh` pulla
   entrambi i modelli se chiamato senza flag. Per Profile B l'LLM cloud
   non serve in locale. Aggiungere lettura di `MEMORYMESH_LLM_PROVIDER` e
   `MEMORYMESH_EMBED_PROVIDER` → skip automatic il modello se il provider
   corrispondente è non-ollama. Defer a quando F2-01 centralizza config.

2. **Reranker model pull** (minor): ARCHITECTURE.md menziona che il
   reranker (`jina-reranker-v2-base-multilingual`) è gestito separatamente
   via `sentence-transformers` cache dir dentro il container api (non
   Ollama). Non fa parte di F1-06. Il pull avviene al primo uso durante
   F3-04b.

3. **Network egress per ollama pull**: `mm_internal` è `internal: true`
   (no egress). Per `ollama pull` serve internet. Il service `ollama` sta
   SOLO su `mm_internal` nel compose. Questo potrebbe impedire il pull su
   compose "puro". **Verifica manuale necessaria** al primo `make up`
   reale: se `ollama pull` fallisce per no egress, aggiungere `mm_ingress`
   al servizio `ollama` (o creare una terza network `mm_egress`).
