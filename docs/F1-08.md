# F1-08 — Backup pg_dump schedulato + smoke test

**Status:** ✅ done — 2026-04-22.

Backup cifrato age + smoke test roundtrip. Il Makefile aveva già target
`backup`/`restore` ma inline SQL fragile; F1-08 estrae in script portabili,
aggiunge retention, smoke test che verifica integrità roundtrip.

## Deliverable

| File | Ruolo |
|------|-------|
| `scripts/backup-pg.sh` | pg_dump → gzip → age encrypted. Idempotente, retention `KEEP_LAST` configurabile. |
| `scripts/restore-pg.sh` | age -d → gunzip → psql. Atomico con `-v ON_ERROR_STOP=1`. |
| `scripts/test-backup.sh` | Roundtrip smoke: seed 3 righe (unicode incluso), backup, DROP, restore, verifica integrità, retention. |
| `Makefile` | `backup`/`restore` delegano ai nuovi script. Nuovo: `backup-check`. |
| `.github/workflows/ci.yml` | Job `backup-smoke` dopo `postgres-smoke`. |

## Flusso produzione

```
pg_dump (via docker compose exec) | gzip -9 | age -r <PUBKEY> > backup-TS.sql.gz.enc
                                                              ^
                                           cifrato con chiave PUBLIC
                                           (admin ha la PRIVATE off-server)
```

Pattern SECURITY.md §6.4:
- `BACKUP_RECIPIENT` = path al file `.age` con chiave PUBBLICA sul server
- Private key in password manager admin, **non** sul server
- Compromesso server = accesso ai backup encrypted ma non leggibili

### Retention

`KEEP_LAST=30` (default) → mantiene gli ultimi 30 backup per timestamp.
Cron giornaliero → 30 giorni di storia. `KEEP_LAST=0` disabilita retention.

### Scheduling (host cron, non sidecar)

MemoryMesh è single-device home server → usare **host crontab**, non un
container cron sidecar (aggiunge complessità senza beneficio reale).

```cron
# ~/.crontab — backup giornaliero alle 03:00 UTC (dopo distillazione)
0 3 * * *  cd /opt/memorymesh && \
  BACKUP_RECIPIENT=/etc/memorymesh/backup-pubkey.age \
  bash scripts/backup-pg.sh \
  >> /var/log/memorymesh/backup.log 2>&1
```

systemd timer alternative in `docs/F1-08.md` (non incluso — dipende dalla
distribuzione target).

## Setup age keys (one-time)

```bash
# Sul server MemoryMesh:
age-keygen -o ~/.mm-backup-privkey.age
age-keygen -y ~/.mm-backup-privkey.age > /etc/memorymesh/backup-pubkey.age

# Trasferisci ~/.mm-backup-privkey.age OFF-SERVER
# (password manager, YubiKey, ecc.). ELIMINA dal server.
shred -u ~/.mm-backup-privkey.age

# La chiave PUBLIC può restare sul server: age è cifratura asimmetrica,
# chi compromette il server può cifrare MA non decifrare.
```

**Requisiti host:**

| OS | Install |
|-----|---------|
| Ubuntu/Debian 23.04+ | `sudo apt install age` |
| macOS | `brew install age` |
| Windows | `scoop install age` o `choco install age` |
| CI Ubuntu runner | `sudo apt-get install -y age` |

## Smoke test — 8 check (8/8 pass)

`scripts/test-backup.sh`:

1. Build immagine age ad-hoc (`alpine + apk add age`) — NO host age required
2. Generazione ephemeral age keypair
3. Boot postgres healthy
4. Seed schema `test_backup.items` con 3 righe (inclusa stringa unicode
   `gamma è úñîcödé`)
5. Shim `age → docker wrapper` in `$PATH` → `backup-pg.sh` usa l'immagine
6. Backup: file `.sql.gz.enc` generato + magic header `age-encryption.org`
7. DROP schema + restore tramite `restore-pg.sh` → 3 righe sopravvissute
   + contenuto byte-per-byte identico (unicode preservato)
8. Retention: secondo backup con `KEEP_LAST=1` rimuove il primo

## Decisioni di design

### 1. Plain SQL + gzip, non `pg_dump --format=custom`

ARCHITECTURE.md originale e Makefile vecchio usavano `--format=custom`.
Ragionamento in F1-08: `custom` format è proprietario, il restore richiede
`pg_restore` (separato da psql) e non è ispezionabile. Plain SQL + gzip:
- restore con `psql` (già disponibile nel container postgres)
- ispezionabile: `age -d ... | gunzip | grep 'CREATE TABLE'`
- compressione comparabile (gzip 20-40% del plain, custom ~30%)
- cross-version: plain SQL riuscirà anche dopo pg_upgrade maggiore

### 2. `--no-owner --no-acl --clean --if-exists`

- `--no-owner --no-acl`: restore funziona anche se il destination DB ha
  user diversi (es. restore su istanza di test con solo `postgres`).
- `--clean --if-exists`: il dump include `DROP ... IF EXISTS` prima di
  `CREATE`. Restore atomic su DB con schema esistente.

### 3. Age docker image per il test (portabilità)

F1-08 test deve girare su Linux, macOS, Windows senza installare age
globalmente. Il test build un'immagine anonima `alpine + apk add age`
una volta, la usa per age-keygen + age (encrypt/decrypt).

Per proxy verso `backup-pg.sh` (che chiama `age` dal PATH), il test
crea un **shim** `age` in un dir temporaneo, anteposto a `$PATH`. Lo
shim esegue `docker run ... $AGE_IMAGE age "$@"`.

Il shim monta il dir chiavi **alla stessa path POSIX** dell'host
(`-v $KEYS_DIR_MOUNT:$KEYS_DIR:ro`), così `age -d -i
/tmp/ephemeral/keys/privkey.age` funziona senza path-rewrite.

### 4. `cygpath -w` per Docker Desktop Windows

Git-Bash su Windows presenta `/tmp/` come tmpfs MSYS dentro
`%TEMP%`. Docker Desktop NON riconosce questi path come mount valide
(usa WSL2 backend). `cygpath -w /tmp/foo` → `C:\Users\...\Temp\foo` che
Docker Desktop sa trattare.

Lo script applica `cygpath -w` se disponibile (Windows), no-op altrove
(Linux/macOS). Pattern generale per i futuri smoke test cross-OS.

### 5. Test script non imposta `MSYS_NO_PATHCONV=1` globalmente

`MSYS_NO_PATHCONV=1` disabilita la traduzione POSIX→Windows dei path
MSYS. **Necessario** per `docker run -v HOST:CONTAINER` quando CONTAINER
è POSIX (`/keys`, `/data`). **Dannoso** per `docker compose --env-file
/tmp/foo` (il path va tradotto).

Il test usa `env MSYS_NO_PATHCONV=1 docker run -v ...` inline, preservando
la traduzione per `docker compose`.

## Gotcha per sviluppatori futuri

1. **`/tmp/` su Windows**: Docker Desktop non monta Git-Bash tmpfs.
   Usare `cygpath -w` per convertire, OPPURE usare path sotto la root
   progetto (il drive è pre-condiviso).
2. **`MSYS_NO_PATHCONV=1` applica solo per chiamata**, non globalmente.
3. **Stesso minuto = stesso filename**: `backup-YYYY-MM-DD-HHMM.sql.gz.enc`.
   Il test dorme 61s fra due backup consecutivi per avere TS distinti;
   alternativa: parametrizzare TS via env (future work).
4. **`pg_dump` vs superuser**: lo script usa `$PG_ADMIN_USER` (default
   `postgres`). Se un giorno si vuole girare `pg_dump` come role non-super,
   serve `GRANT USAGE` su schema + `GRANT SELECT` su tutte le tabelle.
5. **age header check**: il magic `age-encryption.org` è nei primi ~32
   bytes del file. Uso `head -c 32 FILE | grep -q` per validare — niente
   dipendenza da `file` utility.
