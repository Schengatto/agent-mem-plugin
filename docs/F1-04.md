## Decisioni di design

### 1. Raw SQL via `op.execute()`, non `op.create_table()`

Alembic `autogenerate` non gestisce bene:
- HNSW con opzioni `m=16, ef_construction=64`
- GIN su `tsvector GENERATED ALWAYS AS (...) STORED`
- Partial index `WHERE distilled_into IS NULL`
- `INCLUDE (...)` (covering indexes)
- Trigger function PL/pgSQL + CREATE TRIGGER
- ALTER TABLE ENABLE ROW LEVEL SECURITY + CREATE POLICY

Tenere lo SQL in `op.execute()` multilinea mantiene il codice **vicino** allo
schema documentato in `docs/ARCHITECTURE.md §2`. Stesso wording, stessi commenti,
stesso ordering. Il diff è ovvio, la review è un grep.

### 2. Ordering FK-aware

Le FK richiedono creazione in due fasi:

- **Stage A/B/C:** users → admin_users → projects/admin_* → device_keys → …
- **Stage D:** `admin_pair_tokens` referenzia sia `admin_users` che `device_keys`,
  quindi creata dopo entrambe.
- **Chiusura FK circolare `observations.session_id`:** creata come colonna
  plain nel CREATE TABLE di observations (sessions ancora non esiste), poi
  chiusa via `ALTER TABLE ... ADD CONSTRAINT` dopo il CREATE TABLE sessions.

### 3. `mm_admin` come schema owner, non `postgres`

Il target runtime di `make migrate` è `DATABASE_ADMIN_URL=mm_admin`. Motivi:
- non vogliamo esporre la password superuser al container `api`
- `mm_admin` è già il ruolo dell'admin plane (bypasserà RLS come owner → corretto)

Richieste le modifiche a `init-db.sql`:
1. `GRANT CREATE ON SCHEMA public TO mm_admin`
2. `ALTER DEFAULT PRIVILEGES FOR USER mm_admin ...` in aggiunta a
   `FOR USER postgres` — così i grant si applicano sia che Alembic giri come
   postgres (test/bootstrap) sia come mm_admin (runtime).

Nel test `scripts/test-alembic.sh` usiamo `postgres` come user Alembic perché
il test script ha accesso alla password superuser via `PG_ADMIN_PASSWORD`;
questo assicura che lo schema bootstrap funzioni anche senza mm_admin (caso
manuale `psql -U postgres ... alembic upgrade head`).

### 4. Schema opt-in "sempre creato", write opt-in

`manifest_entries_accessed` è la tabella della Strategia 16 avanzata.
`docs/TASKS.md` la descrive come condizionata a `MEMORYMESH_FP_LOGGING=true`.

Implementato come migrazione **sempre applicata** (0002) con scrittura
dall'applicazione gate via `FP_LOGGING_ENABLED`. Alternativa scartata: migrazione
condizionale da env — introduce deriva schema fra ambienti e rompe FK se
qualche migrazione futura la referenzia.

Costo: zero righe quando disabilitata. Beneficio: schema deterministico.

### 5. `uuidv7()` vs `gen_random_uuid()`

Usato `uuidv7()` (pg18 nativo) solo dove l'ordinamento temporale conta:
`admin_sessions.id` (cookie lookup ordinato per last), `admin_pair_tokens.id`
(audit). Altrove `gen_random_uuid()` (random, no leakage di tempo).

### 6. Port mapping nel test — problema Docker internal network

`postgres` è solo su `mm_internal` (`internal: true`). Docker Compose NON pubblica
port mapping per container su reti `internal: true` (non c'è path host-routable).

Fix nel test: l'override compose aggiunge anche `mm_ingress` (bridge non-internal)
SOLO durante il test. In produzione postgres resta isolata.

## Smoke test — 12 verifiche

`scripts/test-alembic.sh` esegue:

1. `alembic upgrade head` su DB pulito
2. Tutte le 19 tabelle applicative esistono
3. Solo tabelle attese + `alembic_version` (niente tabelle spurie)
4. 27 indici critici presenti (HNSW, GIN, partial, composite INCLUDE, LRU)
5. HNSW con `m=16, ef_construction=64`
6. `manifest_entries_scope_path` ha `INCLUDE (obs_id, one_liner, type, priority)`
7. Trigger `single_admin_trg` blocca il 2° INSERT su `admin_users`
8. RLS abilitata su 5 tabelle: observations, vocab_entries, manifest_entries, sessions, manifest_entries_accessed
9. Policy RLS presenti (1+ per tabella)
10. Idempotenza: 2° `upgrade head` = no-op
11. `alembic current` = 0002 (head)
12. `alembic downgrade base` rimuove tutte le tabelle; re-upgrade ripristina

## Come eseguire

```bash
# Locale, richiede Docker Desktop + python3 (>=3.11)
make migrate-check
# oppure direttamente
bash scripts/test-alembic.sh
```

CI gira lo stesso script su ubuntu-latest dopo `postgres-smoke`.

## Note operative

**Se serve ricreare lo schema da zero in dev:**
```bash
docker compose down -v postgres
docker compose up -d postgres
make migrate
```

**Autogenerate futuro:** per ora `target_metadata = MetaData()` vuoto
(`app/db/__init__.py`). Quando in F2-01 arriveranno i modelli SQLAlchemy
(per il typing lato API), `target_metadata` dovrà puntare al Base.metadata
per abilitare `alembic revision --autogenerate`.

**Ruolo runtime migration:** l'`api` container gira come `1000:1000` con
`read_only: true`. Se vogliamo eseguire `alembic upgrade head` durante il
boot dell'api, serve un volume scrivibile per i lock (Alembic usa
`alembic_version` sul DB — ok) e il container deve avere `DATABASE_ADMIN_URL`
(già presente in docker-compose.yml).
